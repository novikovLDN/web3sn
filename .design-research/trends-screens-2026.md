# Тренды 2026–2027 — по экранам и по главной странице

Дополнение к `.design-research/trends-2026.md`. Тот отчёт остаётся базой по
позиционированию, копирайтингу, performance-craft, цвету и текстуре. Здесь —
то, чего в нём нет: **драматургия главной, приёмы «недоступности», подача
каждой из пяти дисциплин, кириллическая типографика и переходы**.

Дата сборки: июль 2026.
Стек проекта: React 18 + Vite + framer-motion 12 + Lenis 1.3.25 + three 0.169 / R3F 8 + Tailwind 3. **GSAP не установлен** — все рекомендации ниже даны так, чтобы работать на этом стеке.

## РАЗДЕЛ 1. Главная страница: «дорого и недоступно»

### 1.1 Механика недоступности

Базовый отчёт §2.6 уже дал главный вывод, и он контринтуитивный, поэтому его
надо повторить перед всем остальным:

> **Ни один сайт из выборки 24 топ-студий не использует датированную доступность.** Ни «currently booking Q3», ни «осталось 2 слота», ни счётчиков, ни зелёной точки «available for work». Ноль.

Причина: датированный дефицит **фальсифицируем и протухает**. Вернувшийся
посетитель, увидевший те же «limited slots» дважды, узнаёт, что его обманули.
**Премиальный дефицит структурен, а не темпорален.**

Эталон формулировки — Obys:

> `"Obys takes on a limited number of projects each year, partnering with marketing leaders and founders who value authorship, clarity and long-term brand impact"`

Четыре механизма в одной фразе: дефицит без числа и даты (нефальсифицируем);
`partnering with`, а не `working for` (симметрия); названа **сеньорность
покупателя** (отсекает тех, кто не подписывает); квалификация **по ценностям,
а не по бюджету** — сказано «если тебе нужно дёшево, это не сюда», ни разу не
сказав «дорого».

**Прямое следствие для этого сайта — три правки, каждая на одну строку:**

1. **Убрать плашку `Открыт для проектов` с зелёной пульсирующей точкой из hero.**
   Это ровно тот сигнал, которого нет ни у одной топ-студии. Базовый отчёт
   §2.6 формулирует выбор без компромисса: **«Соло-разработчик должен выбрать:
   прозрачность доступности (конвертит быстрее, цена ниже) ИЛИ избирательность
   (конвертит медленнее, цена выше). Оба сигнала одновременно невозможны.»**
   Заказчик просит «дорого и недоступно» — значит, плашка уходит. Токены
   `--ok / --ok-bright` после этого можно удалить из палитры: единственный
   цвет вне акцентной температуры исчезает, дисциплина становится абсолютной.
2. **Убрать `CONTACT.availableFrom`** (месяц доступности). Та же причина —
   это датированный дефицит, и `ИНСТРУКЦИЯ.md` уже честно предупреждает, что
   «устаревшая дата хуже отсутствующей». Правильное решение — не обновлять
   дату, а убрать поле.
3. **Добавить одну структурную строку об избирательности** — в `About` или в
   `FAQ`. По модели Obys, на русском, без числа и без даты. Например:
   «Беру ограниченное число проектов в год. Работаю с основателями и
   директорами по маркетингу, для которых важны авторство и длинный горизонт.»

**Уже сделанное правильно — не сломать.** Три вещи на сайте уже работают как
приёмы недоступности, и они сильные:

- **`Портфолио по запросу`** — «Большая часть работ закрыта соглашением о неразглашении и не публикуется». Это лучший из имеющихся сигналов: закрытое портфолио читается как признак уровня клиентов, а не как отсутствие работ. Надо только переставить его выше (Раздел 3-БИС).
- **`Кому вы отказываете`** в FAQ — прямое озвучивание отбора. Редкий и сильный ход. Ответ на этот вопрос должен быть самым конкретным на сайте.
- **Отсутствие цен** и отсутствие выдуманных отзывов (секции честно не рендерятся без данных).

**Что стоит добавить из арсенала, которого пока нет:**

- **Квалификация на форме.** Базовый отчёт §2.6 разбирает механику Cuberto: поле «бюджет проекта» с фиксированными вилками, где минимальная опция — уже высокая. «Сообщение — это пол: все с бюджетом $3k самоотсеиваются до контакта. Форма делает квалификацию, чтобы студии никогда не пришлось говорить "мы дорогие"». Сейчас форма в `ContactSection` — три поля (Имя / Почта / Задача) без квалификации. Добавление вилки бюджета делает работу прайс-страницы без прайс-страницы.
- **Два адреса почты** (общий и по новым проектам) — дешёвый сигнал, что входящих достаточно, чтобы их сортировать.
- **Реквизиты юрлица** в футере. Базовый отчёт §2.8: «Для соло-разработчика это непропорционально мощно: ИНН/номер говорит "я могу выставить счёт вашему отделу закупок", а это и есть реальный блокер на крупных сделках».
- **Год начала практики** — по §2.6 доминирующий долговечный сигнал авторитета: дёшев, вечен, нефальсифицируем. Работает лучше, чем «8 лет опыта», потому что не требует обновления.

**Вербально: минус превосходные степени, плюс преуменьшение.** Тон в
`content.ts` («Сдержанная авторитетность. Дорогой специалист не убеждает —
он излагает») задан верно и совпадает с выводом §2.7. Один приём, которого
пока нет — **самоирония**: «Только тот, кому не нужна работа, может шутить
про работу». Одна такая строка на всю страницу делает больше, чем абзац
о компетенциях.

### 1.2 Первый экран: погружение вместо статичного заголовка

**Что есть сейчас.** `ReactiveGrid` — 2D-канвас, сетка точек `SPACING 46px`,
влияние курсора `RADIUS 190px`, простаивающая синусоида `t += 0.004`, точки у
курсора окрашиваются в акцент. DPR ограничен до 2, rAF-цикл останавливается
`IntersectionObserver`'ом при уходе hero из кадра, `prefers-reduced-motion`
убивает волну.

Это сделано **технически безупречно** и стоит почти ноль. Но как приём это
самый распространённый «интерактивный фон» в вебе — точечная сетка, реагирующая
на курсор, встречается в шаблонах Framer и Webflow сотнями. Заказчик просит
погружения; сетка точек его не даёт.

**Хореография загрузки** сейчас: nav `0.1` → eyebrow `0.25` → `NOVIKOV`
посимвольно `0.35` (`SplitText`, маска из `y: 110%`, `duration.slower 0.95`,
`stagger.letter 0.018`, суммарно ≤0.6s) → statement `0.7` → плашка и кнопки
`0.85` → нижняя панель `1.1`, дисциплины `1.2 + i*0.09`. Итого ~2 секунды
выстроенного входа. **Это хорошо и это надо сохранить** — менять стоит не
тайминг, а то, во что этот тайминг разрешается.

**Три направления, по убыванию отношения эффект/риск:**

**(а) Заменить сетку точек на одну вещь с весом.** Базовый отчёт §4 даёт
формулу выжившей премиальной формы: **«одна твёрдая идея, выполненная чисто —
один объект с настоящим весом, атмосферой и кадрированием, а не нагромождение
эффектов»**, и там же — предупреждение, что тяжёлый WebGL-hero по умолчанию
является обязательством, а не активом (800kB–2MB JS до первого пикселя).

Разрешение противоречия для этого проекта: **three.js уже в бандле** (экраны
услуг и 3D-остров), значит маржинальная стоимость hero-сцены — не рантайм, а
только сама сцена. Условия, при которых это оправдано:
- сцена грузится **после** первого пикселя, поверх уже отрисованного статичного кадра;
- есть честный фолбэк (механика `shaderFailed` из `MotionScreen` уже написана);
- сцена — один объект, а не система частиц.

**(б) Довести существующий канвас, не заменяя его.** Дешёвый путь с высоким
эффектом: сетка точек остаётся, но начинает реагировать не только на позицию
курсора, но и на **скорость** его движения и на скорость скролла. В
`MotionScreen` это уже реализовано — `useVelocity(scrollY)` →
`useTransform([-2500,0,2500],[-5,0,5],{clamp:true})`. Поле, которое
«запаздывает» и «догоняет», ощущается как материя; поле, которое просто
подсвечивается под курсором, ощущается как эффект.

**(в) Заголовок, который реагирует на указатель.** `NOVIKOV` набран `MTS Wide` —
переменного начертания у шрифта нет, поэтому вариативная ось отпадает. Но
доступен трекинг: расстояние между литерами слегка расходится под курсором и
собирается обратно. Пружина `spring.gentle {180, 26, 1}`. Приём тихий,
работает на любом шрифте и не трогает layout, если применять к
`letter-spacing` на отдельных `<span>` с зафиксированной шириной контейнера.

**Про звук.** Аудиогейт на входе (`Enter` / `Enter without audio`, как у Unseen
Studio из §2.2 базового отчёта) — самый сильный приём «церемонии» из
существующих. Но он оправдан **только** вместе с (а): звук на сайте без
тяжёлой сцены читается как претензия. Если hero остаётся лёгким — звука быть
не должно.

**Про прелоадер.** `LoadingScreen` уже есть, и он сделан правильно — прогресс
по реальным сигналам (`document.fonts.ready` = 45, `window.load` = 55), пол
900ms, потолок 1900ms, выход `y: '-100%'` за `duration.slower` с
`ease.editorial`. Базовый отчёт §2.2 предупреждает: «Фейковый прелоадер на
лёгком сайте — junior-tell, издержка без актива за ней». Здесь прогресс
честный, так что претензии нет. Но **пол в 900ms — это уже удержание**:
если ассеты пришли за 300ms, сайт держит посетителя лишние 600ms. Оправдать
это можно ровно одним способом — сделать то, что делает Locomotive: превратить
время загрузки в **высказывание**, которое собирается по фразе. Тогда 900ms
— это не задержка, а предложение, которое нельзя пропустить.
Чего категорически не делать — цикла приветствий `Hello / Bonjour / Guten tag`:
по §2.2 это самый скопированный прелоадер в вебе, и в 2026 он идентифицирует
автора как последователя.

### 1.3 Эмоциональная драматургия длинной страницы

Здесь основная работа, и здесь же самый большой разрыв между текущим сайтом и
задачей. Сейчас страница ритмически **ровная**: после hero каждая секция —
типографика плюс волосяные линии, на одном фоне `--surface`, с одинаковым
вертикальным отступом `--section-y: clamp(5rem, 12vh, 11rem)` и одинаковым
паттерном появления (`Reveal`, `inView { once: true, amount: 0.25 }`).
Дисциплинированно — и монотонно. Единственный температурный сдвиг за всю
страницу — светлая `ServicesSection` (`--n-950`), и он потрачен на самую
скучную секцию.

**Четыре инструмента ритма, в порядке силы:**

**(1) Температура фона как драматургия, а не как разделитель.**
Правило: **одна инверсия на страницу, на самой сильной секции**. Сейчас
инверсия стоит на `Services`. Переставить её на `Projects` — на «Портфолио по
запросу». Тогда самое сильное высказывание сайта («работы закрыты NDA»)
получает единственный за всю страницу удар светом. Всё остальное остаётся
тёмным, и от этого удар работает.

Техника — анимировать **только** `background-color`, по базовому отчёту §6:
«Scroll-driven inversion на границах секций, анимируется только
`background-color`. Дёшево, эффектно, не требует WebGL». Никакого
`backdrop-filter`, никаких смешений.

**(2) Полноэкранные паузы.** Между плотными секциями — экран, на котором почти
ничего нет: одна строка, много воздуха, фон `--temp-deep #060505` (глубже
базового). Пауза не несёт информации; она нужна, чтобы следующая секция
читалась как удар. Три подряд идущие секции одного такта (Process →
Testimonials → FAQ) — это ровно место, куда просится пауза.

Базовый отчёт §1.3 даёт норму воздуха: контент занимает 45–60% ширины
вьюпорта, вертикальные отступы между секциями 12–20vh, не 80px. Текущий
`--section-y` максимум `11rem` (176px) — это **ниже нормы** на больших
экранах. Для секций-пауз нужно `min-height: 100svh`, а не отступ.

**(3) Контраст плотности, а не только размера.** Дорогая длинная страница
чередует: плотный информационный блок → почти пустой экран → плотный.
Сейчас плотность на главной ровная от Hero до Contact. Дешёвый способ создать
контраст — не добавлять, а **вычитать**: убрать `StatsSection` как секцию и
вернуть её одной строкой прозы (Раздел 3-БИС), убрать `TestimonialsSection`
целиком.

**(4) Дыхание вместо однократного появления.** `inView { once: true }` глобально
означает, что страница «включается» один раз и дальше стоит статично. Для
одной-двух ключевых секций стоит завести `{ once: false, amount: 0.4 }` —
секция слегка гаснет, уходя из кадра, и оживает, возвращаясь. Применять
**точечно**: если так ведёт себя вся страница, это читается как AOS-скрипт
(и базовый отчёт §5 честно помечает AOS-подобные раскрытия как
неподтверждённый, но вероятный признак дешевизны).

**Предлагаемая партитура главной:**

```
Hero                 тёмный, движение, ~2s хореография входа        [удар]
Marquee              тёмный, одна строка, быстро                    [выдох]
Projects / NDA       СВЕТЛЫЙ #f7f4ec — единственная инверсия        [удар]
пауза                #060505, одна строка, 100svh                   [тишина]
About                тёмный, портрет, длинный текст                 [плотно]
Services             тёмный, пять строк, интерактив                 [плотно]
Process              тёмный, рельс на скролле                       [средне]
пауза                #060505, 100svh                                [тишина]
Stats-в-строку       тёмный, одна строка прозы                      [быстро]
FAQ                  тёмный, две колонки                            [плотно]
Contact              тёмный, огромная почта, полный вьюпорт         [удар]
```

Базовый отчёт §2.9, дефолт №7: «Футер = полный вьюпорт, огромный email,
локальное время, copy-to-clipboard». Локальное время — сильная деталь: она
тихо отвечает на вопрос часового пояса и одновременно показывает, что сайт
живой.

## РАЗДЕЛ 2. По экранам

**Общий принцип, выведенный из двух эталонных экранов** (обоснование — Раздел 3.3):
экран читается дорого не когда на нём много эффектов, а когда на нём есть
**четыре вещи одновременно**:

1. одно персистентное состояние, которым управляет посетитель, и оно расходится по всему экрану;
2. одно число, посчитанное на странице, а не заявленное словами;
3. собственная типографическая шкала экрана (не только `var(--t-*)`);
4. одна аффорданса, объявленная посетителю прямым текстом.

`WebDesignScreen` и `BrandingScreen` имеют все четыре. `MotionScreen` имеет
полторы, `DevelopmentScreen` — ноль, `Modeling3DScreen` — ноль.

---

### 2.1 Motion-дизайн — `MotionScreen.tsx`

**Текущее состояние.** Технически самый богатый экран: R3F Canvas с
собственным GLSL (`MotionShader.tsx`, uniforms `uTime/uPointer/uPointerDir`,
simplex-шум), детект WebGL с фолбэком `shaderFailed`, кастомный `CursorRing`
поверх канваса (`cursor: none`), `Magnetic` кнопка, sticky-сцена на 280vh,
skew на скролл-скорости через `useVelocity(scrollY)` →
`useTransform([-2500,0,2500],[-5,0,5],{clamp:true})`, `DopeSheet` как навигация.
Палитра «петроль» — `#071316 / #0E2226 / #1F5C63 / #7FB2AE / #E2725B`,
Bricolage Grotesque + Geist Mono.

**Диагноз.** Экран показывает, что автор **умеет** моушн, но не показывает,
что автор его **понимает**. `EasingLab` и `TimingScale` локально интерактивны,
но ничего из того, что посетитель там настроил, не влияет на остальной экран.
Плюс сам файл — 389 строк, и секции 3/6/9/10 (statement, три принципа, большое
слово, CTA) — это статические текстовые блоки. Против пяти полностью
интерактивных секций у `WebDesignScreen`.

**Три приёма, которые стоит внедрить:**

**(а) Глобальная кривая экрана.** Сделать `EasingLab` источником правды:
выбранная посетителем кривая становится CSS-переменной на корне экрана, и на
неё переезжают **все** переходы экрана — появления секций, hover-состояния,
sticky-сцена. Это прямой аналог дока палитры в `BrandingScreen`.

```tsx
// значение из EasingLab пишем в корень экрана
<div style={{ '--screen-ease': `cubic-bezier(${c.join(',')})` } as CSSProperties}>
```

Почему читается дорого: посетитель не смотрит на демо кривых — он **живёт**
внутри выбранной кривой. Разница между «вот график easing» и «весь интерфейс
теперь двигается так» — это разница между иллюстрацией и авторством.

**(б) Живое число.** Сейчас `TimingScale` пересказывает `duration.*`, но
ничего не вычисляет. Базовый отчёт §3c даёт готовый закон, который можно
показать живьём: **постоянная воспринимаемая скорость**, где длительность —
функция пройденного расстояния (<100px → 200ms, 100–500 → 300ms, 500–1000 → 400ms,
полный вьюпорт → 500–600ms). Слайдер расстояния → на экране считается
рекомендованная длительность и тут же проигрывается на реальном объекте.
Это ровно то, что делает `WebDesignScreen` с трекингом и интерлиньяжем.

**(в) Разбор до/после.** Одна секция, где один и тот же переход показан дважды:
слева — с линейной кривой и без стаггера, справа — с `ease.entrance` и
`stagger.item 0.09`. Синхронный запуск по одной кнопке. Компетенция моушн-дизайнера
доказывается **сравнением**, а не демонстрацией; показать плохой вариант рядом
может только тот, кто уверен в хорошем. Механика уже написана в
`WebDesignScreen` (drag-шторка на компенсированном `scaleX`) — переиспользуется.

**Чего избегать:** ещё одного встроенного рила. Vimeo/YouTube-эмбед на экране
моушн-дизайнера в 2026 — это признание, что сам сайт не является демонстрацией.

### 2.2 Разработка — `DevelopmentScreen.tsx`

**Текущее состояние.** Терминал с печатью (16ms/символ, пауза 1400ms),
`BrowserWindow` с превью, `CodeMarquee` **дважды**, шесть sticky-стопкой
карточек этапов (`scale = 1 - (total-1-i)*0.03`), кольцо технологий на 320vh
со скролл-вращением (`angle = useTransform([0,1],[0,TAU])`), фоновые плавающие
глифы `</> { } ( ) ;` на `opacity 0.1`. Палитра «зелёный терминал» —
`#04070a / #0b1117 / #4ade80`, Handjet + JetBrains Mono.

**Диагноз — самый серьёзный из пяти.** Экран показывает **эстетику**
разработки, а не инженерную культуру. Разница принципиальна, и базовый отчёт
§2.8 её формулирует жёстко: «Дизайнерское портфолио оценивается по артефакту;
разработческое — по свидетельствам суждения… любой может заказать красивое
портфолио». Зелёный терминал, плавающие скобки и бегущая строка кода — это
**визуальные цитаты** разработки, то есть ровно то, что заказывают, когда
разработки нет.

Конкретно:
- Ноль собственной типографики: всё через `var(--t-h1)` / `var(--tr-h1)`, ни одного `clamp()`, ни одного решения про интерлиньяж. На фоне специмена Archivo/IBM Plex Mono в `WebDesignScreen` читается как заготовка.
- `STANDARDS` — это проза-утверждения («60 fps», «контраст»), не подтверждённые на странице ничем. Это ровно тот провал, которого избегает `BrandingScreen`, считая WCAG живьём.
- Магические числа вне системы: `speed = 16`ms и пауза `1400` захардкожены (должно быть `duration.cinematic * 1000`), шаг `0.03` в стопке карточек не объяснён.
- Декоративный фон вместо доказательного. У эталонных экранов нет украшений: каждый слой что-то доказывает.
- Две одинаковые `CodeMarquee`.

**Три приёма:**

**(а) Заменить `STANDARDS` на измеренную панель.** Не «60 fps» словами, а живой
счётчик кадров, число draw-call'ов сцены и вес бандла экрана, посчитанные на
странице в момент просмотра. Базовый отчёт §10 даёт готовый инструментарий:
`Long Animation Frames (LoAF)` API и `web-vitals` с attribution-сборкой.
Почему дорого: заявление «я забочусь о производительности» стоит ноль;
счётчик, который посетитель может испортить, открыв тяжёлую вкладку —
это проверяемое утверждение. Плюс он ловит инженерную аудиторию мгновенно.

**(б) Назвать один компромисс.** Базовый отчёт §2.8, сигнал senior №2:
«Названные ограничения и компромиссы, а не названные инструменты… Списки
инструментов ранжируют тебя среди других джунов; заявленные компромиссы
вынимают тебя из сравнения». На экране должен появиться блок формата
«здесь я выбрал X вместо Y, и вот чем заплатил». Для этого сайта есть
готовый честный пример: **Lenis вместо нативного скролла** — куплена
инерция, заплачено конфликтом с `position: sticky`, доступностью и
`scroll-driven animations`; отсюда отключение при `pointer: coarse` и
`prefers-reduced-motion`. Это настоящее решение с настоящей ценой.

**(в) Собственная типографическая шкала + убрать декор.** Явные `clamp()`
с решениями про трекинг и интерлиньяж (с учётом кириллической поправки
`--lh-display: 0.88` из `tokens.css`), плюс удалить `FloatingGlyphs` и
второй `CodeMarquee`. Освободившееся место — под (а) и (б).

**Чего избегать:** логосетки техстека (базовый отчёт §2.8, junior-tell №3),
полосок процентов навыков (junior-tell №7), проектов формы туториала.

### 2.3 3D-моделинг — `Modeling3DScreen.tsx`

**Текущее состояние — самый слабый экран.** 456 строк, пять секций:
hero с ленивым `Model3D` (R3F, `useGLTF` через `import.meta.glob('src/models/*.glb')`,
`ContactShadows`, `Environment` + `Lightformer`, два `useFrame` — турнтейбл и
lerp за курсором), bento-сетка из шести ячеек, шесть шагов пайплайна,
`<dl>` из четырёх позиций, CTA. Палитра «глина» — `#0e0c0a / #181410 / #c9a882`,
Unbounded + Onest.

**Диагноз.** Экран **иллюстрирует** 3D вместо того, чтобы демонстрировать ремесло.
Ни одного слайдера, ни одного тумблера, ни одного до/после, ни одного
переключателя состояния. Из моушна — только `ease.standard`; `ease.entrance`
и `ease.editorial` не используются нигде. Скролл-привязанного движения нет
вообще. Каждая bento-ячейка помечена одним и тем же декоративным ромбом `◆` —
на фоне сконструированного SVG-знака в `BrandingScreen` это плейсхолдер.
И главное: **модель может не отрисоваться вовсе** — она зависит от
пользовательских `.glb`, а фолбэк — голый спиннер. Единственная визуальная
вещь на hero может оказаться пустой.

Плюс bento — сомнительная структура именно здесь (см. Раздел 0): это самая
узнаваемая сетка AI-генерации.

**Три приёма — все про «ремесло, которого не видно на финальном рендере»:**

**(а) Слои пайплайна как персистентное состояние.** Док внизу экрана (прямой
аналог дока палитры в `BrandingScreen`) с переключателем стадий:
`wireframe → retopo → UV → clay → AO → beauty`. Переключение меняет **всё**:
саму модель в hero, превью в сетке, подписи. Почему дорого: финальный рендер
умеет делать генератор картинок; ретопология и развёртка — нет. Показывая
топологию, автор показывает единственную часть работы, которую нельзя
подделать.

**(б) Живые метрики модели.** Не «топология» словом в `<dl>`, а посчитанные
из загруженного GLTF числа: полигонов, вершин, материалов, размер текстур,
плотность текселя, число draw-call'ов. `three.js` отдаёт это из
`renderer.info` и обхода геометрии — считается на клиенте, врать нельзя.
Это прямой аналог живого WCAG-расчёта в `BrandingScreen`.

**(в) Убрать bento, поставить одну модель под свет.** Базовый отчёт §4:
выжившая премиальная форма 3D — «одна твёрдая идея, выполненная чисто:
один объект с настоящим весом, атмосферой и кадрированием, а не
нагромождение эффектов». Референс из того же отчёта — Iventions, где «свет
как основной элемент дизайна, каждый проект подан как объект под точечным
светом». Шесть bento-ячеек с ромбами заменяются на один объект, который
можно вращать, и стадии которого можно переключать.

Плюс обязательное: **фолбэк должен быть не спиннером, а осмысленным
статичным кадром.** Экран не имеет права оказаться пустым.

### 2.4 Брендинг — `BrandingScreen.tsx` (точечные улучшения)

Экран сделан правильно, переделывать нечего. Четыре точечных усиления:

1. **Показать построение знака в движении, а не тумблером.** Сейчас сетка модулей 12×20, зона отступа и направляющая окружность включаются переключателем. ## РАЗДЕЛ 3. Аудит текущего состояния — что уже есть и куда это двигать

### 3.1 Что в коде уже сделано правильно

Система токенов (`src/design/tokens.css`, `src/design/motion.ts`) — сильная,
её не надо переделывать, надо **использовать полнее**:

- Нейтральная шкала уведена в тёплое (`#060505 → #f7f4ec`), не в серое. Это ровно то, что `trends-2026.md` §2.3 называет разницей между «дорогим чёрным» и дефолтным `#000`.
- Один акцент, три светлоты (`--a-dim #c23a17` / `--a #ef4a23` / `--a-bright #ff6a3d`) + альфа-производные. Дисциплина 60/30/10 соблюдена.
- Обратная зависимость трекинга от кегля (`--tr-display -0.045em` → `--tr-body -0.005em`).
- **Комментарий про кириллицу в `--lh-display: 0.88`** — про то, что «й» и «ё» поднимаются выше прописной высоты и срезаются при плотном интерлиньяже. Это редкая по грамотности вещь, её надо распространить на все экраны (сейчас явно учтено только в `WebDesignScreen` и `BrandingScreen`).
- `.optical-left { margin-left: -0.055em }` — оптическая компенсация левого края.
- Дисциплина `will-change` через класс `.animating`, а не постоянно.
- `[data-idle='true']` + `useOffscreenPause` — пауза анимаций вне кадра.

### 3.2 Мёртвый код, который надо оживить

**`.grain` в `tokens.css:237` не используется НИ НА ОДНОМ экране.** Фильтр
`feTurbulence baseFrequency='0.82' numOctaves='3' stitchTiles='stitch'`,
`opacity: 0.035`, `mix-blend-mode: overlay` — рецепт корректный (совпадает с
каноническим из §3.1 базового отчёта). Это готовая общая визуальная подпись,
которая сейчас просто не подключена. См. Раздел 6 про то, где её включать.

### 3.3 Что делает `WebDesignScreen` и `BrandingScreen` эталонными

Четыре свойства, которых нет у трёх остальных экранов:

1. **Персистентное состояние, которым управляет посетитель, и оно расходится по всему экрану.** В `BrandingScreen` — фиксированный док палитры внизу: переключаешь одну из пяти палитр (terra/indigo/pine/plum/coal), и одновременно перекрашиваются знак, свотчи, визитка, аватар, вывеска, обложка. В `WebDesignScreen` — клавиша `G`, включающая 12-колоночный оверлей и 8px-бейзлайн поверх всего экрана.
2. **Числа, посчитанные на странице, а не заявленные.** `BrandingScreen` считает WCAG-контраст живьём (`luminance()` / `contrast()` / `grade()`). `WebDesignScreen` считает трекинг и интерлиньяж как функцию кегля: `tracking = -0.005 - t*0.04`, `leading = 1.62 - t*0.66`, при живом слайдере базового кегля 13–22px и пяти модульных отношениях (1.125 / 1.2 / 1.25 / 1.333 / 1.5).
3. **Собственная типографическая шкала экрана, а не только `var(--t-*)`.** `WebDesignScreen`: Archivo 400–800 + IBM Plex Mono, `h1 clamp(3rem, 12vw, 10.5rem) / -0.05em / 0.95`. `BrandingScreen`: Instrument Serif + Onest, `h1 clamp(3.2rem, 12vw, 9.5rem) / -0.045em / 0.92`. Экран получает собственный голос.
4. **Объявленная посетителю сигнатурная аффорданса.** Клавиша `G`; док палитры. Пользователю прямо показывают, что здесь есть, чем поиграть.

**Это и есть формула, которую надо перенести на три остальных экрана.**
Не «добавить эффектов» — а: одно персистентное состояние + одно живое
вычисление + собственная шкала + одна объявленная аффорданса.

## РАЗДЕЛ 3-БИС. Аудит главной страницы

Текущий порядок секций (`src/App.tsx`), с авторским комментарием-воронкой
«Кто → чем занимаюсь → почему мне можно верить → что получите → доказательство
→ как это устроено → что говорят → возражения → просьба»:

`Hero → Marquee → About → Stats → Services → Projects → Process → Testimonials → FAQ → Contact`

### Что не так со структурой — по мерке базового отчёта

`trends-2026.md` §2.3 установил каноническую премиальную последовательность:
**hero → WORK немедленно → кто мы → услуги коротко → пруфы консолидированно →
футер-как-CTA**, и зафиксировал: «Work перед services — практически
универсально… Services-first — паттерн низкого чека».

**Текущая главная нарушает это дважды:**

1. **Projects стоит шестым, после About, Stats и Services.** Между hero и
   первой работой — четыре секции разговоров о себе. Это ровно инверсия
   премиального паттерна.
2. **Stats (счётчики) стоит третьим** — то есть плотность пруфов вынесена
   наверх. Базовый отчёт §2.3: «Плотность пруфов обратно коррелирует с ценой.
   Уверенность структурно тиха».

**Рекомендуемый порядок:**

```
Hero → Marquee (короткий, как выдох) → Projects/«портфолио по запросу»
  → About → Services → Process → Stats (сжатые, в строку, не счётчики)
  → FAQ → Contact
```

Projects сейчас показывает пустое состояние `Портфолио\nпо запросу` —
и это **не проблема, а актив**, если поставить его вторым экраном. Строка
«Большая часть работ закрыта соглашением о неразглашении и не публикуется» на
позиции №2 работает как приём недоступности (Раздел 1.1). На позиции №6 она
работает как извинение.

### Секции, которые читаются как шаблон

По результатам аудита кода — четыре слабых места, в порядке серьёзности:

| Секция | Проблема | Куда двигать |
|---|---|---|
| **StatsSection** | Четыре count-up счётчика в сетке с волосяными разделителями. Самый распространённый паттерн портфолио в природе. Плюс сами цифры помечены в `content.ts` как `⚠️ ЗАПОЛНИТЬ РЕАЛЬНЫМИ ЗНАЧЕНИЯМИ`. | Убрать счётчики совсем. Базовый отчёт §2.4: «Цифры, а не бейджи… набраны собственным шрифтом сайта». Формат Locomotive — голое число в скобках внутри строки прозы, без анимации: `Awards & Recognitions (295)`. Одна строка вместо секции. |
| **FaqSection** | Учебниковый аккордеон: нумерованные строки, плюс, который поворачивается в крестик. Реализация через `grid-template-rows: 0fr → 1fr` за 450ms — технически отличная, но **форма** абсолютно стандартная. | Форма важнее реализации. Вариант: не аккордеон, а разворот в две колонки — вопрос слева крупно, ответ справа всегда открыт. Возражения, поданные открыто, читаются увереннее, чем спрятанные под плюсик. |

Сейчас секция возвращает `null` (нет реальных данных). | Базовый отчёт §2.4 прямо: «Отзывы почти полностью отсутствуют наверху рынка… логотип клиента И ЕСТЬ отзыв». **Не заполнять. Удалить секцию.** |
| **ProcessSection** | Вертикальный таймлайн с точками и заливающейся линией. Скролл-привязанный градиентный рельс (`useScroll offset ['start 75%','end 65%']` → `useSpring {140,30,0.9}`) — достойный апгрейд, но база всё равно общая. | Оставить, но убрать длительности этапов (`3–5 дней`… `1 неделя`). Названные сроки — сигнал подрядчика, а не автора. |

### Ритмическая проблема

Три подряд идущие секции — Process, Testimonials, FAQ — имеют одинаковый
такт «лейбл + заголовок + список». В исходниках это осознано, и раскладка
варьируется (одна колонка / сетка / асимметричная пара), но **бит один и тот же**.
Лечение — в Разделе 1.3: между ними нужна не другая раскладка, а другая
температура и другая плотность.

### Самая слабая точка всего сайта

**Переход между главной и пятью экранами услуг не анимирован вообще.**
`App.tsx:54–64` — обычный тернарник, без `AnimatePresence`, без exit-анимации,
только `requestAnimationFrame(() => jumpToTarget(0))` и CSS-фейд `screen-in`
`opacity 0→1, 0.5s ease`. В остальном моушн-система выстроена тщательно, и
именно в самый драматургически заряженный момент — когда посетитель выбирает
дисциплину — сайт делает джамп-кат. См. Раздел 4.

### Второе по слабости

**Hero-канвас — единственная сигнатурная графика на всей главной.**
`ReactiveGrid` (2D canvas, `SPACING 46px`, `RADIUS 190px`, DPR clamp до 2,
rAF под `IntersectionObserver`) сделан аккуратно и дёшево. Но после первого
экрана визуальный интерес падает обрывом: всё остальное — типографика плюс
волосяные линии. Это дисциплина, но не погружение. См. Раздел 1.2 и 1.3.

### Мелочи, которые стоит закрыть

- `SERVICES_COPY.subtitle` в `content.ts` не используется: `ServicesSection.tsx:316–319` хардкодит свой вариант. Токен дрейфует.
- 3D-остров в `ContactSection` — самый тяжёлый ассет страницы, стоит между формой и футером и тематически оторван от момента «нанять». Честная подводка `Пока вы здесь` спасает не полностью.
- Портрет в `AboutSection` — SVG-плейсхолдер с подписью «Сюда портретное фото». На живом сайте читается как незаконченность.

## РАЗДЕЛ 4. Переходы между секциями и экранами

Это **самая недоработанная часть проекта** и одновременно самая дешёвая в
исправлении: приёмы ниже не требуют новых зависимостей.

### 4.1 Проблема

`App.tsx:54–64` переключает пять экранов услуг обычным тернарником. Нет
`AnimatePresence`, нет exit-анимации, есть `requestAnimationFrame(() => jumpToTarget(0))`
и CSS-фейд `screen-in` (`opacity 0→1, 0.5s ease`). Роутера нет — это
одностраничное приложение с состоянием `screen`.

Драматургически это худший возможный момент для джамп-ката: посетитель только
что выбрал дисциплину, то есть проявил максимальное намерение за весь визит.

### 4.2 Что применимо на этом стеке

**View Transitions API — не подходит здесь, и это надо зафиксировать.**
Причина не в поддержке браузерами, а в архитектуре: кросс-документные
переходы (`@view-transition { navigation: auto }`) требуют настоящей навигации
между документами, а здесь SPA без роутера. Same-document переходы через
`document.startViewTransition()` технически возможны, но конфликтуют с
`AnimatePresence` (обе системы хотят держать старое и новое дерево
одновременно), и обходятся дороже, чем дают. **Вывод: на этом проекте
переходы делаются на framer-motion, который уже стоит.** Если когда-нибудь
появится роутер — вернуться к вопросу.

**Шторка (curtain) — основной приём, максимум эффекта на минимум кода.**
Механика уже существует в проекте: `LoadingScreen` уходит через `y: '-100%'`
за `duration.slower` (0.95s) с `ease.editorial [0.65,0,0.35,1]`. Тот же
компонент переиспользуется как переход между главной и экраном:

```
1. клик по услуге
2. шторка входит снизу:  y: '100%' → '0%'   340ms  ease.exit [0.7,0,0.84,0]
3. под шторкой: смена screen + jumpToTarget(0)   (0ms, невидимо)
4. шторка уходит вверх:  y: '0%' → '-100%'  620ms  ease.editorial
```

Асимметрия обязательна: вход быстрый и резкий (`ease.exit` — ускоряющаяся
кривая), уход медленный и мягкий (`ease.editorial`). Базовый отчёт §3a даёт
для editorial-шторок диапазон **1000–1400ms** на полный цикл — 340 + 620 = 960ms
плюс кадр на смену состояния попадает точно в него.

Усиление за три строки: на шторке — название выбранной дисциплины,
набранное `--t-h1`, входящее и уходящее вместе со шторкой. Тогда переход не
скрывает смену, а **объявляет** её.

**Морфинг общего элемента (shared element).** Пять строк услуг в
`ServicesSection` — это готовые исходные точки. При клике строка
(`3D-моделинг`, `Разработка`, …) не исчезает, а переезжает в позицию
заголовка целевого экрана. На framer-motion это `layoutId`:

```tsx
// в ServicesSection
<motion.h3 layoutId={`svc-${id}`}>{title}</motion.h3>
// в целевом экране
<motion.h1 layoutId={`svc-${id}`}>{title}</motion.h1>
```

Требует общего `AnimatePresence` и `LayoutGroup` на уровне `App.tsx`.
Почему читается дорого: связь между «я нажал вот на это» и «вот куда я попал»
становится физической, а не логической. Это тот самый приём, из-за которого
переходы у топ-студий описывают как «движения камеры, а не анимацию
элементов» (базовый отчёт §1.2, разбор Uncommon Studio).

Важно: `layout`-анимации framer-motion считаются на главном потоке и могут
дёргаться на длинных списках. Здесь список из пяти строк — безопасно.

**Sticky-сцены.** Уже используются и работают: стопка карточек проектов
(`targetScale = 1 - (total-1-index)*0.03`, `offset: ['start end','start start']`),
шесть карточек этапов в `DevelopmentScreen`, кольцо технологий на 320vh,
горизонтальная галерея на 280vh в `MotionScreen`. Все — на `useScroll` +
`useTransform` поверх `position: sticky`, без GSAP ScrollTrigger. Это
правильный выбор для React: sticky нативен и не ломается при ресайзе.

**Про нативные scroll-driven animations (`animation-timeline: scroll()/view()`).**
Соблазнительно, но здесь **не применять**, по одной конкретной причине:
Lenis работает через трансформацию контейнера, а нативный `scroll()` timeline
привязан к настоящей позиции скролла. Их сочетание даёт рассинхрон.
Проект уже сделал выбор в пользу Lenis (`lerp: 0.085`); значит,
скролл-привязанное движение остаётся на `useScroll` из framer-motion, который
читает то же значение, что и Lenis. Это и есть тот «названный компромисс»,
который стоит вынести на `DevelopmentScreen` (Раздел 2.2б).

### 4.3 Переходы внутри страницы

- **Якорная навигация** уже проходит через `scrollToTarget` (`duration: 1.4`, `easeOutCubic`) — это хорошо и соответствует норме §3a для перемещений на полный вьюпорт.
- **Границы секций.** Сейчас переходы между секциями оформлены скруглением верхних углов (`--r-2xl: clamp(28px, 4vw, 56px)`, у Services 40/50/60px). Базовый отчёт §5 помечает крупные скругления (24px+, особенно 44px) как признак AI-генерации. **Здесь это скругление секции, а не карточки, так что прямого попадания нет** — но 60px близко к границе. Рекомендация: скругление оставить только на единственной инвертированной секции, где оно работает как «лист бумаги, наложенный сверху», и убрать со всех остальных.
- **Курсор.** `CustomCursor` смонтирован глобально, `CursorRing` — локально в `MotionScreen` с `cursor: none`. Базовый отчёт §2.8 (junior-tell №9): «кастомный курсор, ухудшающий юзабилити» — эффект без нагрузки за ним. Проверить: курсор должен что-то **делать** (менять форму над интерактивным, показывать подсказку), а не просто быть кружком-с-задержкой.

## РАЗДЕЛ 5. Кириллическая типографика

**Это главный пробел базового отчёта.** В нём перечислено 60+ гарнитур, и
почти ни одна из них не решает задачу русскоязычного сайта. Apercu, GT America,
Founders Grotesk, Canela, Ogg, Söhne, Editorial New, Migra, Reckless,
Tiempos, Domaine, GT Sectra — **кириллицы либо нет вовсе, либо она есть только
в дорогих расширенных пакетах**, которые надо проверять по конкретной лицензии,
а не по названию семейства.

> ⚠️ Покрытие кириллицы у каждой конкретной гарнитуры **обязательно проверять на
> странице словолитни перед покупкой**. Название семейства ничего не гарантирует:
> у многих зарубежных шрифтов кириллица есть только в отдельном «Cyrillic»
> или «Pro/Extended» пакете, и цена отличается кратно.

### 5.1 Что уже стоит на сайте

**MTS Wide**, самохост, начертания 300/400/500/700, `woff2`, Bold и Regular
в `<link rel=preload>` в `index.html`, `font-synthesis: none`. Одна гарнитура
на всю главную.

**Это сильное решение, и вот почему.** Базовый отчёт §1.3 приводит данные
Typewolf по 40 топовым студийным сайтам: **12 из 40 используют ровно один
шрифт** (Wade and Leta — Söhne, Morrow — Untitled Sans, Cast Iron Design —
Unica77). Одна гарнитура на весь сайт — это статусный жест, а не экономия.
Плюс `font-synthesis: none` — редкая по грамотности деталь: браузер не имеет
права подделывать жирность и курсив, а поддельный «болд» кириллицы выглядит
особенно плохо.

Плюс MTS Wide решает главную проблему сразу: **кириллица у него родная**,
не адаптированная, с нормальными пропорциями.

**Риск позиционирования.** MTS Wide — корпоративный шрифт конкретного бренда.
Аудитория, знакомая с российским рынком, его узнаёт, и на портфолио
независимого специалиста это может прочитаться как «взял чужой фирменный
шрифт». Проверить лицензию на коммерческое использование вне бренда —
обязательно, до всего остального.

### 5.2 Куда двигать типографику главной

Формула из базового отчёта §1.3, адаптированная под кириллицу:

```
display (акцидентный)  +  neo-grotesque (текст)  +  mono (метаданные)
```

Mono здесь — это код-сигнал: он говорит «я разработчик» без единого слова.
На главной сейчас mono-начертания нет вообще (`JetBrains Mono` подключён,
но заскоплен на экран моушна), при том что `.t-mono` и `--tr-mono: 0.32em`
в токенах определены. Это ещё один кусок мёртвого CSS.

**Кандидаты с гарантированной кириллицей** (все проверяемы, большинство
бесплатны):

*Гротески для текста:*
- **Onest** — уже используется в `BrandingScreen` и `Modeling3DScreen`. Open-source, кириллица родная, переменный. Один из немногих свободных шрифтов, спроектированных с кириллицей как первичной письменностью, а не как добавкой. Хороший дефолт для текста.
- **Golos Text** — сделан для «Госуслуг», кириллица родная, свободный. Нейтральный до безликости — это его достоинство в роли текстового.
- **Involve / Rothek / TT-семейства** (TypeType) — платные, кириллица родная, большой выбор ширин.
- **Suisse Int'l Cyrillic**, **Graphik Cyrillic** (Commercial Type) — платные, дорогие, и именно они читаются как «настоящие» на русскоязычном премиальном рынке.
- **Inter** имеет отличную кириллицу — но базовый отчёт §5 фиксирует, что Inter как *единственная* гарнитура каталогизирована как признак AI-генерации. Как fallback в стеке — нормально; как основной выбор — нет.

*Акцидентные / display:*
- **Unbounded** — уже используется в `Modeling3DScreen`. Кириллица родная, переменный, свободный. Заметный характер; на всю страницу тяжеловат, для заголовков — работает.
- **Bounded** — тот же автор, более сдержанный.
- **Кириллические словолитни**: **type.today**, **CSTM Fonts**, **Brownfox**, **Contrast Foundry**, **Rentafont**. Это правильное место для покупки акцидентного шрифта под русский премиум — там кириллица первична, а не портирована. Конкретную гарнитуру выбирать глазами, под уже принятую тёплую палитру.

*Mono с кириллицей — узкое место:*
- **JetBrains Mono** — кириллица есть, свободный, уже в проекте.
- **IBM Plex Mono** — кириллица есть, свободный.
- **Geist Mono** — базовый отчёт хвалит его как шрифт для dev-портфолио, **но кириллическое покрытие надо проверить перед использованием в русских подписях**. Если кириллицы нет — использовать только для латинских техметок (`fps`, `webgl`, `glb`), что вообще-то допустимо и даже уместно.

### 5.3 Что проверить в текущих экранах

Экраны услуг подключают свои гарнитуры, и часть из них по кириллице
проблемна:

| Экран | Гарнитуры | Статус |
|---|---|---|
| `WebDesignScreen` | Archivo 400–800 + IBM Plex Mono | Кириллица есть у обеих. Безопасно. |
| `BrandingScreen` | **Instrument Serif** + Onest | ⚠️ Два вопроса. Первый — кириллица Instrument Serif под вопросом, проверить. Второй — базовый отчёт §5 каталогизирует Instrument Serif как перегруженный AI-слопом. Заменить на кириллический editorial-антиквенный из type.today / CSTM. |
| `MotionScreen` | **Bricolage Grotesque** + Geist Mono | ⚠️ Кириллицу обеих проверить. |
| `DevelopmentScreen` | **Handjet** + JetBrains Mono | ⚠️ Handjet — пиксельный/вариативный акцидентный, кириллица у него есть, но как основной шрифт экрана он читается как игровой, а не инженерный. Не соответствует задаче «показать инженерную культуру» (Раздел 2.2). Заменить. |
| `Modeling3DScreen` | Unbounded + Onest | Кириллица есть у обеих. Безопасно. |

**Отдельно — кириллическая метрика.** В `tokens.css:87–92` уже есть верный
комментарий: «й» и «ё» поднимаются выше прописной высоты, и при плотном
интерлиньяже верхняя строка срезает диакритику; `0.88` — минимум. Явно этот
фикс применён только в `WebDesignScreen` (`lineHeight 0.95` с комментарием)
и `BrandingScreen` (`0.92`). **Три остальных экрана его не документируют**, а
`MotionScreen` использует `0.88` в hero — то есть ровно на границе. Проверить
на реальных заголовках с «й» и «ё».

Второй кириллический нюанс, которого нет нигде: **разрядка прописных**.
`--tr-label: 0.18em` и `--tr-mono: 0.32em` заданы одинаково для латиницы и
кириллицы. Кириллические прописные шире латинских и требуют **меньшей**
разрядки при том же оптическом результате. Для русских лейблов
`0.14–0.16em` вместо `0.18em`.

## РАЗДЕЛ 6. Текстура, палитра, движение — конкретные значения

Базовый отчёт покрывает палитры (§2.6, восемь наборов) и рецепт зерна (§3.1).
Здесь — только то, что применимо **к этому проекту**, с учётом уже принятых
токенов.

### 6.1 Палитра: что менять и что не трогать

**Не трогать.** Текущая система — тёплая нейтральная шкала `#060505 → #f7f4ec`
плюс одна акцентная температура `#c23a17 / #ef4a23 / #ff6a3d` — уже
соответствует всему, что базовый отчёт называет дорогим: near-black не чистый
чёрный (§2.3), один акцент (§2.6, правило 60/30/10), тёплый сдвиг вместо
нейтрального серого.

**Что добавить — вторую температуру для драматургии, не для декора.**
Сейчас на главной **одна** светлая секция (`ServicesSection`, `--n-950 #f7f4ec`,
скруглённая сверху на 40/50/60px). Это единственный температурный удар на
странице, и он стоит не в том месте — на услугах, самой скучной секции.

Рабочий набор из трёх температур для длинной страницы:

```
--temp-deep:  #060505   /* провал: полноэкранные паузы, перед ударом */
--temp-base:  #0c0b0a   /* дефолт страницы */
--temp-lift:  #141211   /* приподнято: секции-доказательства */
--temp-flip:  #f7f4ec   /* инверсия: ровно один раз за страницу */
```

Правило: `--temp-flip` использовать **один раз** и на самой сильной секции.
Второе появление обесценивает первое.

**Чего избегать (по §5 базового отчёта, всё подтверждено):** фиолетово-синие
градиенты, свечения цветной тенью на тёмном, стеклянные карточки на сетке,
градиентный текст. К этому проекту это отношения не имеет — палитра чистая.

### 6.2 Зерно — включить то, что уже написано

`.grain` (`tokens.css:237`) **не подключён ни на одном из пяти экранов услуг**.
На главной он есть (Hero, Process, Testimonials, FAQ, Contact), на экранах — нет.
Это единственная общая визуальная подпись сайта, и она наполовину выключена.

Текущие значения корректны и совпадают с каноническим рецептом базового отчёта §3.1:

```css
baseFrequency='0.82'  numOctaves='3'  stitchTiles='stitch'
opacity: 0.035        mix-blend-mode: overlay
```

Поправки к применению:

- `0.035` — нижняя граница рабочего диапазона. Базовый отчёт §3.1 фиксирует, что на **плоских тёмных поверхностях** продакшн-значения выше: `0.10–0.15`, потому что там зерно выполняет работу по устранению 8-битного бэндинга. Диапазон `0.02–0.08` — это для зерна **поверх изображений**. Рекомендация: держать `0.035` глобально, но поднимать до `0.09–0.12` на секциях с градиентной подложкой (Hero-канвас, радиальные свечения на `MotionScreen` и `Modeling3DScreen`).
- `numOctaves='3'` — правильный потолок, каждая октава примерно удваивает стоимость фильтра.
- **Никогда не анимировать** `baseFrequency` или `seed` — это форсирует перезапуск фильтра каждый кадр. Если нужен «живой» grain, базовый отчёт даёт единственно верный путь: предрендерить 3–5 кадров шума и переключать их CSS-анимацией `steps()`, ноль JS на кадр.
- `feTurbulence` — стоимость на **растеризации**, не на кадре. Статичное зерно бесплатно после отрисовки.

### 6.3 Хроматическая аберрация и дизеринг — дозировка

Базовый отчёт §3.6 покрывает аберрацию, §3.4 — halftone/dither. Практическое
правило для этого сайта:

- **Хроматическая аберрация** — только на движении и только на одном элементе за раз. RGB-сплит на статичном заголовке читается как пресет 2019 года. Уместное применение: сдвиг каналов на 0.5–1.5px в момент перехода между экранами (Раздел 4), затухающий к нулю за `--d-base` (400ms). В `MotionScreen` уже есть шейдер с simplex-шумом — аберрацию логично встроить туда как uniform, а не как CSS-фильтр.
- **Дизеринг / 1-bit** — сильный приём для `Modeling3DScreen`: показать превью модели дизерингом, а по наведению — чистый рендер. Это буквально «ремесло, которого не видно на финале» (Раздел 2.3). В остальных местах не использовать.
- **Галоид / полутон** — на этом сайте не нужен. Тёплая палитра + зерно уже дают материальность; полутон добавит ретро-читаемость, которая противоречит позиционированию.

### 6.4 Движение — что уже есть и чего не хватает

Система в `src/design/motion.ts` полная и корректная:

```
ease.standard   [0.32, 0.72, 0, 1]     — интерфейсный дефолт
ease.entrance   [0.16, 1, 0.3, 1]      — рабочая лошадь появлений (expo-out)
ease.exit       [0.7, 0, 0.84, 0]      — уходы
ease.editorial  [0.65, 0, 0.35, 1]     — шторки, крупные полотна
ease.soft       [0.4, 0, 0.2, 1]
ease.overshoot  [0.34, 1.56, 0.64, 1]

duration: instant .12 / fast .24 / base .4 / slow .65 / slower .95 / cinematic 1.4
spring:   cursor {520,40,0.6} / snappy {380,30,0.8} / gentle {180,26,1}
          heavy {90,22,1.4} / scroll {140,30,0.9, restDelta .0005}
stagger:  letter .018 / word .045 / line .075 / item .09 / block .14
inView:   { once: true, amount: 0.25, margin: '0px 0px -12% 0px' }
```

Сверка с таблицами базового отчёта §3a/§3b — расхождений нет, значения лежат
внутри рекомендованных диапазонов. `stagger.letter 0.018` точно попадает в
«15–20ms, достаточно быстро, чтобы читаться одним жестом». Ограничение
`SplitText` на `MAX_TOTAL = 0.6s` реализует правило бюджета стаггера
(`items × stagger + duration ≤ ~1.2s`) — это сделано грамотно.

**Три пробела:**

1. **Выходы не быстрее входов.** Базовый отчёт §3a: «exits should be ~30–40% shorter than entrances, because the user has already decided». Сейчас `ease.exit` определён, но отдельной шкалы длительностей для выходов нет. Добавить:
   ```ts
   duration.exitBase = 0.26   // против base 0.4
   duration.exitSlow = 0.42   // против slow 0.65
   ```
2. **`ease.overshoot [0.34, 1.56, 0.64, 1]` — опасный токен.** Базовый отчёт §4 (каталог AI-slop) прямо помечает «bounce and elastic easing» как признак сгенерированного интерфейса. Различие, которое там же проводится: пружина с честной массой/натяжением — нормально, декоративный overshoot, прикрученный к кнопке — тель. Рекомендация: `ease.overshoot` не использовать на главной вообще; оставить только внутри `MotionScreen`, где демонстрация кривых — это и есть контент.
3. **`inView.once: true` глобально.** Для длинной эмоциональной страницы это правильно на появлениях, но блокирует приёмы «дыхания» из Раздела 1.3 (секция, которая гаснет при уходе из кадра). Нужен второй пресет:
   ```ts
   inViewLive = { once: false, amount: 0.4 }
   ```

### 6.5 Lenis — текущая настройка

`src/hooks/useSmoothScroll.ts`: `lerp: 0.085`, `wheelMultiplier: 0.9`,
`touchMultiplier: 1.2`, `smoothWheel: true`; выключается при
`prefers-reduced-motion` и при `(pointer: coarse)`; нативный `scroll-behavior`
принудительно `auto`, чтобы не конфликтовать. Это соответствует
«minimum-harm configuration» из §6.3 базового отчёта — менять не нужно.

`lerp: 0.085` — заметно тяжелее дефолтного `0.1`. Для «дорогого» ощущения это
правильный сдвиг: вес скролла читается как вес сайта. Ниже `0.07` уходить не
стоит — начинается ощущение задержки ввода.
