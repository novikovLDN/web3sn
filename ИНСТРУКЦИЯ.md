# Инструкция по редактированию сайта NOVIKOV.

Здесь простыми словами — как менять контент без знания кода. Почти всё правится
в **двух файлах**: `src/data/content.ts` (весь текст) и `src/data/projects.ts`
(кейсы). После правок нужно один раз пересобрать сайт (см. раздел 8).

---

## 0. Что нужно заполнить в первую очередь

Сайт спроектирован под дорогое позиционирование, но часть данных — заглушки,
которые я не мог заполнить за вас. **Пока они не заменены, сайт работает
вполсилы.** По убыванию важности:

| Что | Где | Почему это важно |
|-----|-----|------------------|
| **Настоящие кейсы** | `src/data/projects.ts` | Пустое портфолио обесценивает всё остальное. Клиент с бюджетом смотрит на работы. |
| **Реальные цифры** | `STATS` в `src/data/content.ts` | Сейчас там мои оценки (8 лет, 90+ проектов, команда 6). Неверная цифра подрывает доверие ко всем остальным. |
| **Отзывы** | `TESTIMONIALS` в `content.ts` | Секция **не показывается**, пока нет настоящих. Так и задумано. |
| **Логотипы клиентов** | `CLIENTS` в `content.ts` | Тоже скрыто, пока список пуст. |
| **Портретное фото** | секция «Обо мне» | Сейчас там рамка-плейсхолдер с подписью. |
| **Месяц доступности** | `CONTACT.availableFrom` | Устаревшая дата хуже отсутствующей — обновляйте. |

### Почему отзывы и клиенты не выдуманы

Я намеренно не сочинил ни имён, ни компаний, ни цитат. Фальшивая рекомендация
от несуществующего человека проверяется за минуту и стоит дороже, чем
отсутствие отзывов вообще. Обе секции написаны так, что просто **не
отрисовываются**, пока в них нет реальных данных, — сайт от этого не ломается.

## 1. Весь текст сайта

```
src/data/content.ts
```

Один файл на весь копирайт: первый экран, «Обо мне», процесс, вопросы,
контакты. В шапке файла описан **тон голоса** — держитесь его, когда правите:

- конкретика вместо превосходных степеней;
- ни одного восклицательного знака;
- только те цифры, которые можно подтвердить.

Что где лежит: `IDENTITY` (имя, роль, дисциплины) · `HERO` (первый экран) ·
`ABOUT` · `STATS` · `PROCESS` (5 этапов) · `TESTIMONIALS` · `CLIENTS` ·
`FAQ` · `CONTACT`.

## 2. Кейсы (секция «Проекты»)

```
src/data/projects.ts
```

Список `PROJECTS`. Каждый кейс — блок в фигурных скобках `{ ... }`.
Порядок в списке = порядок на странице.

### Ключевое поле — `filled`

| Значение | Что происходит |
|----------|----------------|
| `filled: false` | Кейс **не показывается** |
| `filled: true` | Кейс выводится полностью |

Если ни одного `filled: true` нет, секция показывает достойный блок
**«портфолио по запросу»** — это нормальный приём у специалистов, работающих
под NDA. Никаких «скоро здесь будет магия».

### Поля кейса

| Поле | Что это | Пример |
|------|---------|--------|
| `filled` | Показывать ли кейс | `true` |
| `client` | Клиент или компания | `'Solaris'` |
| `year` | Год | `'2025'` |
| `role` | **Ваша роль** — важно для позиционирования уровня | `'Ведущий дизайнер, команда 4 человека'` |
| `title` | Название кейса | `'Ребрендинг финтех-платформы'` |
| `disciplines` | Список дисциплин | `['Брендинг', '3D', 'Веб']` |
| `challenge` | Задача — что было не так | |
| `solution` | Решение — что вы сделали | |
| `outcome` | Результат: `{ metric, text }` | `{ metric: '+38%', text: 'к конверсии' }` |
| `images` | Картинки: `{ src, alt }` | `[{ src: '/projects/case-1.jpg', alt: 'Главный экран' }]` |
| `link` | Ссылка на живой проект | `'https://solaris.example'` |

> `outcome` необязателен. **Отсутствующая цифра лучше выдуманной** — если
> результат не измеряли, просто не заполняйте это поле.

### Как добавить кейс

1. Положите картинки в `public/projects/`.
2. Заполните блок в `PROJECTS` и поставьте `filled: true`.

### Требования к картинкам

- **Формат:** `.jpg` или `.webp` (легче).
- **Ширина:** 1600–2400 px, вес до ~1.5 МБ.
- **Пропорции:** кадрируйте под **16:10** — секция жёстко держит это
  соотношение, иначе картинку обрежет не так, как вы ожидали.
- **Имя файла:** латиницей, без пробелов: `solaris-brand.jpg`.
- **`alt`** заполняйте осмысленно: это и доступность, и поиск.

## 3. Аудио / плейлист (мини-плеер справа вверху)

**Кидайте аудиофайлы в папку `src/tracks/`** — плеер сам их подхватит.

- Форматы: `.mp3`, `.ogg`, `.wav`, `.m4a`, `.flac`.
- Порядок — по имени файла. Нумеруйте: `01-name.mp3`, `02-name.mp3`.
- Автозапуск после загрузки; если браузер блокирует — трек включится при
  первом клике или прокрутке.

---

## 4. Своя 3D-модель (экран услуги «3D-моделинг»)

Положите файл в папку `src/models/` (например `model.glb`).

- **Формат:** `.glb`. Экспорт из Blender: `File → Export → glTF 2.0 (.glb)`.
- **Анимация:** если есть — проиграется в цикле. Если нет — модель будет
  вращаться на «поворотном столе».
- **Размер/центр:** подгонять не нужно, модель центрируется сама.
- **Вес:** желательно до ~8–10 МБ. Для тяжёлых включите сжатие Draco.
- Берётся первый файл по алфавиту. Пока файла нет — показывается заглушка.

## 6. Контакты и почта

- Почта — переменная окружения `VITE_CONTACT_EMAIL`, фолбэк в коде.
- Тексты секции — `CONTACT` в `content.ts`.
- Соцсети — в `src/sections/ContactSection.tsx`. **Не добавляйте ссылки-заглушки
  `#`**: мёртвая ссылка читается как небрежность. Лучше меньше, но рабочих.

## 7. Дизайн-система (если хотите поменять стиль)

```
src/design/tokens.css   — цвет, типографика, отступы, радиусы, кривые
src/design/motion.ts    — кривые, длительности, ритм появления
```

Это **единственные** места, где заданы цвет и размеры. Поменяете акцент
`--a` в одном месте — он сменится на всём сайте, включая экраны услуг.

Правило проекта: в компонентах не должно появляться сырых цветов вида
`#ef4a23` — только `var(--a)`, `var(--n-700)` и т.п. Иначе система
рассыпается, и сайт снова начинает выглядеть как набор случайных решений.

## 8. Как применить изменения (пересборка)

```bash
npm install      # только один раз, при первой установке
npm run build    # собрать сайт
npm run preview  # посмотреть локально перед деплоем
```

Деплой на **Railway** — просто запушьте в GitHub, сборка запустится сама:

```bash
git add -A
git commit -m "обновил кейсы"
git push
```

> Если `npm` не находится — на этом компьютере Node лежит отдельно, выполните
> сначала: `export PATH="$HOME/.local/node/bin:$PATH"`

> **Важно:** сборка = `tsc && vite build`. Любая опечатка в TypeScript валит
> деплой. Если `npm run build` прошёл локально — пройдёт и на Railway.
